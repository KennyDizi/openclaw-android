# eval

기능 구현 또는 버그 수정 전에 "성공이란 무엇인가"를 먼저 정의하고, 구현 후 체크리스트 기반으로 채점한다.

## 철학

- 성공 기준을 **구현 전에** 정의한다 (테스트 주도의 상위 개념)
- 채점 결과를 커밋 메시지 또는 PR 본문에 포함하여 추적성을 확보한다
- 코드 기반 자동 판정을 우선하고, 관찰 기반 판정은 보조로 사용한다
- **양 버전 동작 필수**: Termux+스크립트와 앱+스크립트 양쪽에서 동작해야 성공이다

## 절차

### Step 1 — Eval 정의

기능/버그 설명을 받아 성공 기준 체크리스트를 작성한다:

**기능적 요구사항** — 동작해야 하는 것:
- [ ] (구체적 동작 기술)
- [ ] Termux 설치 경로에서 동작 확인
- [ ] 앱 설치 경로에서 동작 확인

**비기능적 요구사항** — 성능, 호환성 등:
- [ ] (측정 가능한 기준)

**회귀 방지** — 깨지면 안 되는 것:
- [ ] (기존 기능 보존 확인)
- [ ] 기존 설치 사용자의 업데이트 경로 (update-core.sh) 정상 동작

각 항목에 채점 방식을 태깅한다:
- `[code]` — shellcheck/빌드/테스트 결과로 자동 판정
- `[observe]` — 실행 결과 관찰로 판정 (실기기 테스트 포함)
- `[human]` — 사용자 확인 필요

### Step 2 — 구현

기능 구현 또는 버그 수정을 진행한다. eval 정의를 의식하며 구현.

### Step 3 — 채점

체크리스트 항목별로 PASS/FAIL 판정:

- `[code]` 항목: verify 스킬(shellcheck, 플로우 무결성, 버전 일관성)로 자동 판정
- `[observe]` 항목: 실행 결과를 관찰하여 판정. 실기기 테스트 시 update-test 스킬 활용
- `[human]` 항목: 사용자에게 에스컬레이션

### Step 4 — 리포트 + 기록

채점 결과를 구조화된 리포트로 출력한다:

```
EVAL REPORT: {feature-name}
============================

기능적 요구사항:
  - 항목 1: PASS [code]
  - Termux 경로 동작: PASS [observe]
  - 앱 경로 동작: PASS [observe]

비기능적 요구사항:
  - 항목 1: PASS [code]

회귀 방지:
  - 항목 1: PASS [code]
  - 업데이트 경로: PENDING [human]

종합: X/Y PASS, Z PENDING
```

리포트를 커밋 메시지 body에 포함한다 (제목이 아닌 본문).

## 판정 규칙

- `[code]` 항목 중 하나라도 FAIL → 구현 미완. 수정 필요
- `[observe]` 항목 FAIL → 원인 분석 후 수정
- `[human]` 항목 PENDING → 사용자에게 확인 요청, 블로커 아님
- **양 버전 동작 항목 FAIL** → 반드시 수정. 한쪽만 동작하는 것은 미완
- 전체 PASS (PENDING 제외) → 구현 완료

## 다른 스킬과의 연동

### verify 스킬 연동

eval의 `[code]` 항목 채점에 verify 스킬의 결과를 활용한다:
- shellcheck → 스크립트 문법 검증
- Install/Update 플로우 무결성 → 경로별 동작 검증
- Delivery Path Parity → 양 버전 동기화 검증

### update-test 스킬 연동

eval의 `[observe]` 항목 중 실기기 테스트가 필요한 경우:
- update-test 스킬로 실제 디바이스에서 동작 확인
- 유형 A (기존 설치 업데이트) / 유형 C (신규 설치) 구분

## 주의 사항

- eval 정의는 구현 **전에** 작성한다. 구현 후 작성하면 확인 편향이 생긴다
- 체크리스트는 검증 가능해야 한다. "잘 동작함" 같은 모호한 기준 금지
- 작은 버그 수정에는 eval이 과도할 수 있다. 복잡한 기능/수정에 적용
